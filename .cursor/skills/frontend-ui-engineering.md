# Frontend UI Engineering

## Overview

Build production-quality user interfaces that are accessible, performant, and visually polished. The goal is UI that looks like it was built by a design-aware engineer — not like it was generated by an AI.

## Component Architecture

- Colocate everything related to a component
- Prefer composition over configuration
- Keep components focused
- Separate data fetching from presentation

## State Management

Choose the simplest approach: Local state → Lifted state → Context → URL state → Server state → Global store

## Design System Adherence

### Avoid the AI Aesthetic

| AI Default | Production Quality |
|---|---|
| Purple/indigo everything | Use the project's actual color palette |
| Excessive gradients | Flat or subtle gradients matching the design system |
| Rounded everything | Consistent border-radius from the design system |
| Generic hero sections | Content-first layouts |
| Oversized padding | Consistent spacing scale |
| Stock card grids | Purpose-driven layouts |
| Shadow-heavy design | Subtle or no shadows |

## Accessibility (WCAG 2.1 AA)

- Keyboard Navigation
- ARIA Labels
- Focus Management
- Meaningful Empty and Error States

## Responsive Design

Design for mobile first, then expand. Test at 320px, 768px, 1024px, 1440px.

## Loading and Transitions

Skeleton loading, optimistic updates for perceived speed.

## Common Rationalizations

| Rationalization | Reality |
|---|---|
| "Accessibility is a nice-to-have" | It's a legal requirement and engineering quality standard. |
| "We'll make it responsive later" | Retrofitting is 3x harder. |
| "The AI aesthetic is fine for now" | It signals low quality. |

## Verification

- [ ] Component renders without console errors
- [ ] All interactive elements keyboard accessible
- [ ] Screen reader can convey content
- [ ] Responsive at 320px, 768px, 1024px, 1440px
- [ ] Loading, error, and empty states handled
- [ ] Follows project's design system
- [ ] No accessibility warnings
